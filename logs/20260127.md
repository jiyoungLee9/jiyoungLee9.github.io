# (2026-01-27) storybook 도입
<br>
github Copilot ai 점검 후, 분석 생성 결과를 보다 claude code 연결하여 생성하는 부분까지 알아보고 완벽하게 현타가 와서,<br>
이후 방향성에 대한 전략을 어디로 세워야 하는 고민까지 생김.<br>
그러다 si산업전반의 vibecoding을 포함한 모든 ai시장에서의 현 상황까지 대화해보니 생각했던 그대로가 대답이 나온다.<br><br>

요구사항정의 및 작업결과의 취사선택, 품질처리는 사용자의 몫이지만,<br>
반복적이거나, 체계회돤 업무는 ai를 활용한다. 명령과 결과를 사용자에 의해 품질이 달라지는 것을 감안하더라도,<br>
중위권 수준의, 즉 적당히 보여줄 수 있는 수준의 UI는 ai가 더 정확하고 빠르게 작업한다.<br>
대신 도메인 특수성이 강한 영역은 ai 학습으로 진행하기 매우 어렵다고 판단된다.<br>
하지만, 그마저도 10년내 정리되지 않을까 하는 시선.<br><br>

뭐, 돌고돌아 결국 그래도 기술을 배우는 것 자체가 의미가 있으니 다시 정신차리고 시작한다.<br><br>

애초에 stoybook으로 컴포넌트 관리를 생각해서 라우팅 전 물어보니, 즉각 시작하자고 학습로드맵을 변경한다.<br>
실무에서 특정 library 수준이라도 도입하려면 생태계를 파악하고 document를 돌려 읽느라 바쁘지만,<br>
ai는 그런거 없다. 그냥 설치하고 진행하자고 한다.<br>
npx 통해서 storybook 설치한다. (Playwright은 미설치 상태)<br>
로컬에서 바로 작성된 컴포넌트 확인됨. 상태값(state)에 따른 디자인이 필요하다는 것을 깨달음<br>
app.jsx를 정의가 난무했다면 이제부터는, storybook만 정의한다.<br><br>

규칙이 보이기 시작한 단계다.<br>
react프로젝트의 ui구성에 따른 활용이 레거시와 다른 지점이 보이기 시작한다.<br>
특히 이러한 구조라면 컴포넌트를 어떻게 쪼개어 정리하는 것이 프로젝트의 관건이 될 것이라 보여진다.<br><br>

현재 상태까지 진행하고 나니 전략적인 방향이 보인다.<br>
레거시를 spa전환 프로젝트일 때 어떤 과정이 필요하고, 어떤 점을 유념해야 하는지도 보인다.<br>
마지막으로 디자인의 토큰관리까지 간다면, 소위 차세대로환사업 중 UI설계를 실무적으로 어떻게 풀어서 정의할 수 있다.<br>
실제 프로젝트로 진행할 수 있는 상황을 만들어봐야겠네<br>
