# (2026-02-11) 반복적인 리팩토링
<br>
modal (dialog) 작성하면서 기존에 작성된 컴포넌트 재사용을 하면서 리팩토링의 반복이 시작되었다.<br>
생성된UX(피그마)와 연동될 data를 예상해서 생각하는 각종 props, 또는 css className 등을 다시 설계해서 리팩토링 하는 경우가 빈번해 졌다.<br><br>

- 전역으로 추가될 css용 className : mgt10 등 강제 적용 용, <br>
- 같은 컴포넌트지만 비슷한 명칭의 className의 다중 선언 처리 : 버튼의 primary, secondary, size 처리 등 <br>
- 상황에 따른 추가 Attributes : 조건에 따른 특정 className 추가나, props로만 전달되지 않는 특정 Data Attributes 처리.<br>
- storybook을 통한 ui event 구현 : ui용 템플릿 함수 별도 생성 후 props값에 따른 작동 확인 등.<br><br>

염두해야 될 것들이 자꾸 보이고, 자꾸 다시 만지다보니 고작 12개 컴포넌트인데도 속도가 안붙는다. (한달 넘었는데 겨우 7개 완성이라니..)<br>
이 속도 부분은 어쩌면 ai가 제시하는 jsx스타일이 실무에 확 들어맞지 않는 경우가 많아서 일 수도 있다.<br><br>

레거시 프로젝트는 대부분 개발팀과 커뮤니케이션으로 처리하고, <br>
어지간하면 className 으로 처리하는 부분을 ai는 처음에는 모두 inline style을 처리하려 한다.<br>
단순히 inline style 처리를 하다보니 처음에는 가장 쉬워보이지만, <br>
실제 운영을 하다보면 inline style보다는 className 을 통한 관리가 훨씬 간결하고 선언 역시 편하기에 실무에서 개발팀과 협약하여 쓰는 편인데<br>
ai가 처음 생성한 코드는 가장 쉬운 방법으로 만들고, <br>
그것을 내가 이리저리 조건에 맞게 고친 후, 에러 여부를 확인 하고 ai에게 코드리뷰 받거나<br>
애매한 부분은 ai에게 실무에서 가장 많이 쓰이는 방식으로 업데이트 요청을 하는 방식으로 고쳐 쓰고 있다.<br><br>

뭐 애초에 UI컴포넌트 설계 자체가 한번에 짠 하는 경우가 없다보니, 작성한 컴포넌트를 이후 조금 더 리팩토링, 리팩토링, 또 리팩토링을 거치면서 진행하는데,<br>
가장 큰 어려움은 레거시에서 사용하는 내 방식이 실제 react 실무 프로젝트에도 맞는지 확인하느라<br>
ai에게 지속적으로 확인하는 경우가 많아진다는 것.<br>
